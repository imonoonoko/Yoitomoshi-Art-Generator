# Prompt Meaning Enrichment Reliability Roadmap

Updated: 2026-05-26

Status: Implementation roadmap based on `7_best_practices_research_20260526.md`, the existing requirements handoff, and the current `scripts/enrich-prompt-dictionary-meanings.cjs` implementation.

## 1. Roadmap Goal

Make `dictionary:enrich:meanings` safe enough to run against a wider Prompt Daijiten staging set without turning weak external matches into trusted Japanese meanings.

The implementation must treat external APIs as evidence, not truth. The resulting behavior should be deterministic, auditable, and conservative by default.

## 2. Resolved Policy Decisions

- Evidence hierarchy: `Danbooru exact wiki > Wikidata filtered entity > Wiktionary support > Civitai usage evidence`.
- Default broad `--apply` threshold: `--min-confidence 0.80`.
- Civitai evidence is usage/model-linkage evidence only. It must not become semantic meaning by itself.
- Danbooru redirect-only wiki bodies remain below apply threshold until redirect target resolution is implemented.
- Wikidata hits that look like albums, songs, games, TV series, journals, sculptures, books, organizations, persons, or places are rejected or downgraded for generic prompt tokens.
- `curated` translation jobs are protected from automatic overwrite unless `--force` is explicit.
- Raw provider payloads stay in local staging/cache only. Runtime dictionary output stores generated Japanese text and source attribution, not copied external prose.
- Wikimedia-family requests must use a meaningful User-Agent, keep concurrency at or below 3, and respect `Retry-After` on `429` / `503`.

## 3. Current Baseline

Already present:

- Provider lookups: `lookupDanbooru`, `lookupWikidata`, `lookupWiktionary`, `lookupCivitai`.
- Suggestion creation: `buildSuggestion`.
- Apply path: `applySuggestion`.
- Cache path: `fetchCachedJson`.
- Basic `curated` overwrite guard in `applySuggestion`.
- Meaning lookup staging table in `resources/prompt-dictionary/ingest-schema.sql`.
- Existing dry-run/apply wrapper through `npm.cmd run dictionary:enrich:meanings`.

Missing before wide use:

- Normalized `decision` object with reason and counters.
- `--min-confidence`, `--json-out`, `--fixture-dir`, `--no-network`, `--timeout-ms`, and retry-related flags.
- Explicit `BEGIN IMMEDIATE` / `COMMIT` / `ROLLBACK` around apply batches.
- Offline fixture regression command.
- Provider schema validation and payload hashes.
- Timeout, retry/backoff, `Retry-After`, and provider circuit breaker.
- Clear separation between provider cache records and per-candidate enrichment decisions.

## 4. Milestone P0: Safe Decision And Apply Core

Purpose: Ensure the script can decide, explain, and apply only defensible updates.

Implementation order:

1. Add normalized evidence records before scoring.
   - Required fields: `provider`, `providerKey`, `sourceUrl`, `confidence`, `evidenceKind`, `title`, `summary`, `warnings`, `payloadHash`.
   - Evidence kinds should distinguish `exact-wiki`, `redirect-wiki`, `tag-only`, `wikidata-entity`, `dictionary-definition`, and `usage-only`.
2. Add `scoreEvidence()` and `buildDecision()`.
   - Output shape: `tag`, `candidateId`, `decision`, `reason`, `confidence`, `selectedProvider`, `sourceUrl`, `jaLabel`, `jaMeaning`, `evidence`, `warnings`.
   - Allowed decisions: `apply`, `skip`, `preview-only`, `provider-error`.
   - Required reasons: `high-confidence-danbooru-wiki`, `low-confidence`, `curated-protected`, `ambiguous-provider-result`, `redirect-only`, `usage-evidence-only`, `no-candidate`, `provider-disabled`, `schema-invalid`.
3. Add `--min-confidence`.
   - Dry-run still shows all evidence.
   - Apply uses `0.80` by default.
   - `0.70` to `0.79` becomes `preview-only` unless the user explicitly lowers the threshold.
4. Route `buildSuggestion()` through the decision object.
   - `Civitai` can produce usage notes for review but must not produce semantic source-derived meaning by itself.
   - Danbooru non-redirect exact wiki can become `source-derived`.
   - Wikidata can become `source-derived` only after filtered entity checks.
5. Wrap every apply run in an explicit transaction.
   - `BEGIN IMMEDIATE` before writes.
   - `COMMIT` after all writes succeed.
   - `ROLLBACK` on exception.
6. Expand output counters.
   - Include `applied`, `previewOnly`, `skippedLowConfidence`, `skippedCurated`, `skippedAmbiguous`, `providerErrors`, `providerDisabled`, `cacheHits`, and `cacheWrites`.
7. Add audit storage.
   - Keep raw payloads in `meaning_lookup_cache`.
   - Prefer a separate staging table such as `meaning_enrichment_decisions` for per-candidate `decision`, `reason`, selected provider, confidence, source URL, payload hash, generated text summary, script version, and timestamp.

Exit criteria:

- `node --check scripts/enrich-prompt-dictionary-meanings.cjs` passes.
- `npm.cmd run dictionary:enrich:meanings -- --tag bad_hands --dry-run --provider danbooru` shows an apply-capable Danbooru wiki decision.
- `npm.cmd run dictionary:enrich:meanings -- --tag newest --dry-run --provider wikidata` does not produce an apply decision.
- `npm.cmd run dictionary:enrich:meanings -- --tag 1girl --dry-run --provider civitai` treats Civitai as usage evidence only.
- Temporary-data-root apply smoke shows `curated` rows are not overwritten without `--force`.

## 5. Milestone P1: Offline Fixtures And Regression Command

Purpose: Make provider scoring testable without live network behavior.

Implementation order:

1. Add fixture loading.
   - Flags: `--fixture-dir <path>` and `--no-network`.
   - Suggested path: `scripts/fixtures/prompt-dictionary-meanings/`.
   - Fixture lookup key: provider plus provider key, for example `danbooru-wiki/bad_hands.json`.
2. Add stable fixture cases.
   - `bad_hands`: Danbooru exact wiki accepted.
   - `hands`: generic term accepted only with strong exact source.
   - `hand`: redirect/general-term case does not over-apply.
   - `kimono`: common clothing/object accepted when evidence is exact.
   - `newest`: work-title or noisy Wikidata result skipped.
   - `1girl`: prompt convention/usage evidence, not ordinary dictionary meaning.
   - `style`: ambiguous generic token skipped unless exact prompt-context source exists.
   - Internal QA tag: excluded from automatic batch selection.
3. Add `dictionary:enrich:meanings:test`.
   - Prefer Node's built-in test runner if no project-specific test harness exists.
   - Strip timestamps and volatile URLs before snapshot comparisons.
4. Add `--json-out <path>` for durable reports.
   - Use this for dry-run comparisons and handoff artifacts.

Exit criteria:

- `npm.cmd run dictionary:enrich:meanings:test` passes offline with `--no-network`.
- Fixture decisions are stable across two consecutive runs.
- JSON reports include all counters and per-tag decisions.
- `git diff --check` passes.

## 6. Milestone P2: Provider Reliability And Rate-Limit Policy

Purpose: Make live API use respectful and predictable.

Implementation order:

1. Replace direct `fetch()` calls with `fetchJsonWithPolicy(provider, url, options)`.
2. Add timeout support.
   - Flag: `--timeout-ms`, default `8000`.
   - Use `AbortSignal.timeout()` where available.
3. Add retry/backoff policy.
   - Flag: `--retry`, default `2`.
   - Retry transient failures only: `408`, `409`, `425`, `429`, `500`, `502`, `503`, `504`, timeout, connection reset.
   - Do not retry deterministic failures: `400`, `401`, `403`, `404`, schema invalid.
4. Respect `Retry-After`.
   - Required for Wikimedia-family `429` / `503`.
   - Fall back to jittered exponential backoff with at least a 5-second base delay.
5. Add provider circuit breaker.
   - Flag/default: `--max-errors-per-provider 5`.
   - When tripped, mark remaining provider lookups as `provider-disabled` and continue other providers.
6. Add concurrency control.
   - Wikimedia-family default max concurrency: `3`.
   - Other providers: conservative default `2` to `4`.

Exit criteria:

- Live smoke tests for one tag per provider complete without uncontrolled retry loops.
- Provider error counts are visible in JSON output.
- `Retry-After` behavior is covered by fixture or mocked test.

## 7. Milestone P3: Review Workflow And Runtime Integration

Purpose: Turn conservative decisions into a practical curation workflow.

Implementation order:

1. Export `preview-only` and skipped-but-useful records for human review.
2. Add import/apply support for reviewed records without marking them `curated` automatically.
3. Rebuild runtime dictionary only from promoted/generated Japanese summaries and source attribution.
4. Add Dictionary tab affordances later only after the CLI evidence flow is stable.
5. Consider optional LLM Japanese cleanup only after deterministic source selection succeeds.

Exit criteria:

- Curator can inspect low-confidence candidates without reading raw provider payloads.
- Runtime DB contains generated Japanese summaries and attribution, not raw external text.
- Existing Prompt Daijiten search/autocomplete behavior remains stable after rebuild.

## 8. Wide Apply Gate

Do not run broad `--apply` until all checks pass:

1. `node --check scripts/enrich-prompt-dictionary-meanings.cjs`
2. `npm.cmd run dictionary:enrich:meanings:test`
3. Temporary-data-root apply smoke with rollback verification.
4. `npm.cmd run dictionary:ingest:init -- --data-root output/prompt-dictionary-ingest-reliability-smoke --json`
5. `npm.cmd run typecheck`
6. `git diff --check`

Allowed before this gate:

- Single-tag dry-runs.
- Fixture/offline runs.
- Temporary-data-root apply smoke.
- Manual review of JSON reports.

Not allowed before this gate:

- Broad `--apply` against the normal `userdata` ingest DB.
- Marking automated results as `curated`.
- Shipping raw external payloads in `resources/prompt-dictionary/prompt-dictionary.sqlite`.

## 9. Recommended First Implementation Slice

Implement P0 and the minimal P1 fixture harness together:

1. Add `buildDecision()` and `--min-confidence`.
2. Protect apply with explicit transaction and counters.
3. Add fixture read path for `bad_hands`, `newest`, `kimono`, and `1girl`.
4. Add `dictionary:enrich:meanings:test`.
5. Verify with `node --check`, fixture test, temporary-data-root apply smoke, and `git diff --check`.

This slice is the smallest useful boundary because confidence scoring without fixtures is hard to trust, and fixtures without the decision object do not test the real production behavior.
