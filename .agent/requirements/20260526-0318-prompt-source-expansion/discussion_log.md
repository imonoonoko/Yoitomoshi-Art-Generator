# Discussion Log: Prompt Source Expansion

## 2026-05-26 Initial Request

User wants to massively expand the Prompt Daijiten database by collecting prompts and tags from AI illustration sites, separating prompt tags, and adding Japanese translation/meaning.

Explicitly requested skills:

- `define-requirements`
- `orchestrate-skills`
- `codex-brain`

Skill stack selected:

- `orchestrate-skills`: choose only the relevant skill stack and avoid missing source/legal research.
- `codex-brain`: read Brain hot/index and query existing Yoitomoshi prompt dictionary context.
- `define-requirements`: create durable `.agent/requirements/` handoff before implementation.
- `yoitomoshi-ai-tool-research`: judge external prompt/data sources for Yoitomoshi fit, license/ToS risk, and smallest safe adoption path.

Current repo facts:

- Working tree was clean at start.
- Existing Prompt Daijiten uses `resources/prompt-dictionary/prompt-dictionary.sqlite` generated by `scripts/build-prompt-dictionary-db.cjs`.
- Existing search path is `api.promptDictionary.search` -> `electron/prompt-dictionary.ts` -> SQLite/FTS5 with YAML fallback.
- `docs/maps/02-prompt-management-flow.md` states Prompt Daijiten must not search all data in renderer and must keep source provenance separated.

External evidence checked:

- Civitai public image API sample: `https://civitai.com/api/v1/images?limit=1&nsfw=false&withMeta=true` returned item metadata with `meta.prompt` and `meta.negativePrompt`.
- Civitai API docs wiki now points to `developer.civitai.com`; old wiki is deprecated.
- Civitai Terms of Service prohibit generic crawler/data-mining tools outside provided search agents/tools, so implementation must use official/public API only and avoid HTML scraping.
- Gelbooru DAPI has a documented post endpoint and 100 post/request hard limit, but Gelbooru TOS says not to use automated processes to retrieve/index site contents; bulk automated Gelbooru ingestion is out of MVP.
- DiffusionDB on Hugging Face is CC0, exposes prompt metadata, and can be accessed text-only through Parquet instead of downloading images.
- Hugging Face Dataset Viewer API supports listing rows, searching/filtering, and Parquet access for datasets where viewer support is available.

Decision:

- Build a source-governed ingestion pipeline, not an unrestricted scraper.
- Use API-first and dataset-first ingestion; HTML scraping, Cloudflare bypassing, login/session scraping, and image downloading are non-goals.
- Keep raw imported prompts in local `userdata/` staging, and only promote normalized, deduped, source-attributed tag entries into the shipped SQLite dictionary.

## 2026-05-26 Civitai Importer Execution

Implemented and executed the first real source importer.

- Added `scripts/import-civitai-prompt-dictionary.*`.
- Import mode: Civitai public `/api/v1/images`, `nsfw=false`, `withMeta=true`, no image bytes, no HTML scraping.
- Added prompt tokenization, LoRA/resource separation, adult-token review guard, heuristic Japanese draft labels/meanings, and candidate auto-accept by evidence count.
- Connected accepted staging candidates into `scripts/build-prompt-dictionary-db.cjs`.
- Runtime dictionary grew from 4007 entries to 4242 entries.
- Civitai-derived promoted runtime entries: 245.
- Staging after run: 135 raw metadata records, 1204 candidate tags, 300 accepted candidates.
- Exported accepted candidates to `resources/prompt-dictionary/promoted-candidates.civitai.json` so the enriched runtime DB can be rebuilt without committing raw prompts or depending on ignored `userdata/` state.
- Civitai rate limit surfaced as HTTP 429 during larger runs; importer now has 429 backoff and default source rate lowered to 0.2rps. Continue future imports in small, slow batches.
