1. 全体結論

Prompt系は統合UIへ寄せる

TagComplete、Prompt All in ONE、Better Prompt、Easy Prompt Selectorは、全部が「promptを文字列ではなくタグ単位で扱う」方向を向いている。 Yoitomoshi側では既にタグチップ編集と英語タグ化を始めたので、次はドラッグ並び替え、無効化、履歴、ブラックリスト、My Prompt相当の登録を一体化する。

Model系はRecipe化する

Model Preset Manager、Config Presets、Civitai Helperは、モデルごとの最適なprompt、negative、VAE、解像度、sampler、trigger wordsを保存する思想が強い。 YoitomoshiのModel LibraryとWorkspaceをつなぎ、モデル単位の「推奨Recipeカード」にするのが最短で効く。

画像入力はタグ・補修・構図へ流す

WD14 Tagger、ADetailer、ControlNet、Ultimate Upscaleは、画像を解析してタグ化し、必要な場所だけ補修し、構図を保って高解像度化する流れを作れる。 Drop画像からPrompt Helper、ControlNet Builder、ADetailer、Upscale比較へ連結する導線を強化する。

ライセンス上の注意: ADetailerとForgeはAGPL、Ultimate UpscaleはGPL、Model Preset Manager / Civitai Helper / WD14 TaggerはGitHubメタデータ上で明示ライセンスなし。 将来公開を考える場合、コード移植ではなく、公開READMEから分かる機能要件をYoitomoshi側で独自実装する方針を維持する。

3. Forge同梱拡張群の扱い

API専用起動で使うべき中核

Forge本体はStable Diffusion WebUI上の開発基盤で、資源管理、推論高速化、実験機能を目的としている。 現在のYoitomoshiではGradio UIを開かず、Forge APIとalwayson_scripts/script_argsを直接使う。

sd_forge_controlnet sd_forge_multidiffusion sd_forge_dynamic_thresholding sd_forge_freeu sd_forge_lora SwinIR / ScuNET

Yoitomoshi側で強化する入口

Forge同梱のpreprocessor群はControlNet Builderから使うのが最適。normalbae、tile、reference、inpaint、recolor、revision、marigoldなどは、 「用途カード」と「画像プレビュー」を増やせばユーザーが直接理解できる。

5. 既存機能とのシナジーマップ

Prompt Library / Prompt Helper

  • TagCompleteのタグ辞書を検索候補へ追加する。
  • Prompt All in ONEの履歴・お気に入り・blacklistを、YoitomoshiのPrompt Recipeとして保存する。
  • Better PromptのMy Promptを、Prompt Libraryのユーザーカテゴリに統合する。

LoRA / Civitai / Model Library

  • LoRA trigger wordsをタグ補完とModel Recipeに流す。
  • Civitai preview画像とmetadataをModel Libraryのカードに表示する。
  • 履歴でよく使ったLoRAをモデル別にスコアリングし、Recipe候補へ昇格する。

ControlNet Builder

  • WD14タグ抽出結果から、pose/depth/lineart/reference/tileの候補を出す。
  • Forge同梱preprocessorを用途カードとして整理する。
  • preprocessor出力をWorkspace参照として保存し、再実行なしで復元する。

History / Workspace / Upscale

  • 生成時promptだけでなく「展開前prompt」「ランダムseed」「採用Recipe」をmetadataに残す。
  • Upscale比較結果をHistoryラベルと連動させ、採用候補/素材化済みに分類する。
  • Workspace保存時にModel RecipeとPrompt Recipeをまとめて再現可能にする。

6. 実装しない/慎重に扱う案

案理由代替
Prompt All in ONEを再有効化してGradio UIで使う YoitomoshiはForge API専用起動で、Gradio UI系拡張は起動遅延・互換エラーの原因になる。 機能思想だけをReact側に実装する。
AGPL/GPL拡張のコードを直接移植する 将来公開時のライセンス影響が大きい。 公開READMEから分かる仕様を参考に独自実装する。Forge API呼び出しは既存連携の範囲で使う。
外部翻訳APIを必須にする ローカル創作ツールとして、APIキー・通信・料金・レート制限が障害になる。 ローカル辞書を既定にし、任意で翻訳プロバイダを追加可能にする。
