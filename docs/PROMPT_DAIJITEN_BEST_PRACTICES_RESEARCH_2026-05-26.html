Prompt大辞典 ベストプラクティス追加調査

Prompt Daijiten のソース拡張、意味付け、レビュー導線を大規模化する前に、2026-05-26時点の一次情報と現在の実装状態を照合した調査メモ。 結論は「API/ライセンス/出典をsource registryで先に固定し、raw promptはstagingに隔離し、意味付けは強い根拠だけ自動適用、弱い根拠はレビュー待ちにする」。

確認日: 2026-05-26 対象: Prompt Daijiten / source expansion / meaning enrichment 依頼ID: 019e5fb4-9667-7b83-98b8-9129df21e5fe 一次情報優先

Executive Verdict

今の土台は正しい

`resources/prompt-dictionary/sources.json`、`userdata/prompt-dictionary/ingest.sqlite`、 `promotion_decisions`、`meaning_lookup_cache`、SQLite FTS5 runtime DBという分離は、大規模化に向いた方向。 特にCivitai由来raw promptを同梱DBへ入れず、昇格済みタグだけをruntime辞書へ渡す設計は維持する。

次の弱点は自動意味付け

現状の `dictionary:enrich:meanings` は provider evidence を集められるが、閾値、skip reason、ambiguous判定、curated保護、fixture回帰がまだ薄い。 ここを先に固めないと、数万語化した時に「もっともらしい誤訳」が辞書品質を下げる。

自動収集先は増やしすぎない

追加すべき順は `local-user-prompts`、Civitai public API、Danbooru系tag/wiki metadata、DiffusionDB text-only。 Gelbooru bulk post importやHTML scraping系サイトは、APIがあっても利用条件が弱いのでMVPではdisabledのままにする。

Evidence Ranking

自動意味付けはproviderを横並びにせず、下の優先順位で決める。重要なのは「最も情報量が多いsource」ではなく「このtagの意味として誤適用しにくいsource」を勝たせること。

Rank Evidence Auto Policy Notes
1 curated / user override never overwrite 既存の日本語label/meaningがcuratedならprovider結果は補足表示だけ。
2 Danbooru exact tag + non-redirect wiki apply if safe `General`かつadult_level 0を優先。redirect-onlyはskipまたはreview。
3 Danbooru exact tag without useful wiki preview / review categoryとpost_countは強いが、意味説明としては薄い。
4 Wikidata exact generic concept apply only above threshold garment、body part、material、objectなどは有効。work-title-likeはskip。
5 Wiktionary definition machine-draft only 一般語確認には使えるが、画像生成タグの意味とはズレやすい。
6 Civitai tag search evidence-only モデル/LoRA探索の手掛かり。タグ意味の正本にはしない。
7 Local/user prompt frequency rank / review 「自分が使う語」として重要だが、意味説明は自動で断定しない。

Implementation Best Practices

1. Meaning enrichment should return a decision object

`buildSuggestion()` は `suggestion | null` ではなく、必ずdecisionを返す形にする。 これによりdry-run、apply、Workbench UI、fixture regressionが同じ説明を使える。

Field Purpose
decisionapply / skip / preview-only
reasonhigh-confidence、low-confidence、ambiguous-provider-result、curated-protected、redirect-only、provider-error、no-candidate
confidenceprovider固有scoreではなく、最終decision用のnormalized score
winningEvidenceprovider、sourceUrl、providerKey、category、postCount、entityIdなど
audit採用しなかったevidenceと理由。後からUIで説明できるように残す

2. Provider thresholds should be asymmetric

Danbooru exact wikiとWikidata entity searchは同じconfidenceで扱わない。 Wikidataは一般概念に強いが、同名の作品・曲・人物へ吸われやすい。Civitai tagsは使用例の存在確認であって意味説明ではない。

Provider Case Suggested threshold Action
Danbooru exact wiki, non-redirect, General0.90+auto apply if uncurated and SFW
Danbooru redirect-only0.55skip with redirect-only until target resolution exists
Wikidata exact generic concept with useful JA/EN description0.75+preview/apply only if no stronger evidence
Wikidata likely work/person/fictional characternegative scoreskip with ambiguous-provider-result
Wiktionary noun/adjective definition0.45-0.55machine-draft only
Civitai tag match0.30rank/search evidence only; no automatic meaning overwrite

3. Importers need source snapshots and kill switches

`source_registry_snapshot` はすでに良い。次は各runに「実行時のterms URL、checkedAt、rateLimit、raw storage policy、query defaults」を残し、 source側の規約やAPIが変わった時に過去のimportを無効化できるようにする。

  • source registryに policyStatus: enabled / manual-only / disabled / needs-review を持たせる。
  • CLIは disabled sourceを即error、manual-only は --confirm-source-policy がない限りerrorにする。
  • Workbenchでは sourceごとの「最近のimport」「error」「terms checked date」「raw prompt保存可否」を見せる。
  • sourceをdisabledにしても過去のruntime辞書から該当sourceだけ除外してrebuildできるようにする。

4. Keep raw/staging and shipped/runtime databases separate

現在の設計通り、raw prompt、source payload、cache、review queueは `userdata/` に置く。 `resources/prompt-dictionary/prompt-dictionary.sqlite` は検索に必要な正規化済みtag、短い日本語説明、provenanceだけを含める。

維持する境界: `raw_prompt_records.raw_json` と `meaning_lookup_cache.payload_json` はstaging側。 runtime側へ入れるのは `dictionary_entries`、`dictionary_text`、`dictionary_aliases`、`dictionary_sources`、FTS index。 第三者のprompt本文やwiki本文をそのまま同梱しない。

5. SQLite search should stay rebuild-first

SQLite FTS5のexternal-content tableは容量を節約できるが、content tableとの同期責任が増える。 Yoitomoshiのruntime辞書はbuild scriptで一括生成するので、当面はcontentful FTS + trigram FTSでよい。 数十万語まで増えた時だけ、external-content化またはsearch shard化を検討する。

  • 英語tag検索: `unicode61 remove_diacritics 1 tokenchars '_-'` とprefix searchを維持。
  • 日本語検索: CJK segmentationに頼らず、現在のtrigram FTSを維持。label/meaning/aliasをまとめたsearch_textを正規化する。
  • staging DB: WALでよいが、backupやcopy時は `.db` だけでなくWAL状態に注意。runtime build artifactはDELETE journalで配布向き。
  • `node:sqlite` は公式Nodeでもrelease candidate扱いなので、Electron同梱Nodeのversion固定と `ELECTRON_RUN_AS_NODE=1` 経由の回帰を続ける。

6. Electron IPC must remain narrow

Dictionary Workbenchが増えてもrendererからDB pathやSQLを直接渡さない。 Electron公式docsはcontext isolationと安全なcontextBridge wrapperを推奨しているため、今の `api.promptDictionary.*` のような狭いIPCを維持する。

  • rendererは `listCandidates(filters)`、`savePromotionDecision(payload)`、`inspectIngest()` のような意図ベースAPIだけ呼ぶ。
  • main側でfilter enum、limit上限、source id、decision enumを検証する。
  • Workbenchからimport/rebuildを実行する場合はpreview countを表示し、明示buttonで開始する。
  • 外部URLを開く導線は `shell.openExternal` へ集約し、source URLをallowlistまたはURL parserで検証する。
