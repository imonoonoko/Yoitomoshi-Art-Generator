制作ループ

Yoitomoshi Forge Studioが今つないでいる、日常制作の流れ。

1. 設計する Prompt Composer Slots、タグライブラリ、Research Workflow、モデル別Prompt Profileで作りたい絵の条件を組む。
2. 生成する txt2img / img2img / Inpaint / ControlNet / ADetailer / LoRA / Dynamic PromptをForge APIへ送る。
3. 選別する 履歴、Candidate Board、favorite / candidate / rejected / asset、Pro Recipe Reviewで候補を残す。
4. 仕上げる Upscale Simple / Diffusion / Ultimate、Tile ControlNet、denoise比較、Videoへの派生を使う。
5. 再利用する .yoitoart Workspace、PNG metadata、Model Library、Civitai Recipeヒント、Promptタグ辞書へ制作条件を戻す。

生成コア

txt2img / img2imgの現在の契約。

Generate実行 active tabで決定

endpointは入力画像の有無ではなく、現在のワークスペースタブで決まる。txt2img上では古いinputImageがstoreに残っていてもtxt2imgを実行する。

img2img / Inpaint 入力必須

img2imgはinputImage必須。maskがある場合はinpaint requestとしてForgeへ送る。画像貼り付け時は必要に応じてimg2imgへ自動切替する。

残るリスクと次の焦点

今すぐ壊れているというより、次の判断で見落としやすいところ。

GitHub運用 再利用用

  • GitHubは公開プロダクト配布ではなく、ソース保管、再セットアップ、別環境への再利用を目的にする。
  • runtime/、userdata/、モデル、生成物、秘密情報はGitHubに含めない。
  • コード署名、更新メタデータ、正式アイコン、Mac/Linux対応は、将来必要になった時だけ検討する。
  • NSFW生成モデレーションはアプリ本体にはない。Civitai検索フィルタとは別問題。

制作機能 伸びしろ

  • Phase 6のReference Board / Optional Backendsは設計確認のみで任意フェーズ。
  • Workspace参照保存はできるが、欠落参照の差し替えUIと相対パス化は検討候補。
  • Format ConverterのLoRA抽出系、Inpaint maskの編集履歴/反転/ぼかしは後続候補。

起動性能 Forge側が主コスト

  • rendererは軽い。直近平均でも約228ms。
  • Forge readyはサンプルにより約17.0秒から51.1秒まで幅がある。
  • Toolsの個人環境ヘルスチェックから、Python/Torch、拡張、ControlNet、checkpoint/VAE、API readiness polling の遅延シグナルを分けて確認できる。
  • 安全復旧はsettings正規化、stale download整理、Model Library復旧に限定し、残プロセス停止や孤立partial削除は手動確認に残す。
  • ControlNetの遅延有効化は再起動影響が大きく、現状は常時ロード維持が妥当。

作業ツリー 大きい差分

  • 現在のリポジトリは多数の未コミット変更を含む。
  • 次の実装前には、今回のHTMLレポート以外の既存差分がどの作業に属するか確認した方がよい。
  • 生成物やruntime/userdataを不用意に整形・削除しない。
