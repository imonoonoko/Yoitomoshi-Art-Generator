Yoitomoshi タグライブラリ強化リサーチ

作成日: 2026-05-22 / 対象: Yoitomoshi Art Generator Forge版 / 参考: imonoonoko/Tag-Manager

結論: 次の強化は、タグ追加ボタン単体の改善ではなく、 タグを「chip一覧」から「モデル別・出典付き・alias対応のプロンプト知識ベース」へ拡張する方向が最も効果的です。 Tag-Manager の思想は有用ですが、Tkinter/SQLite UIを移植するより、Yoitomoshiの既存React/Electron構造に合わせて段階的に取り込むべきです。

要約

最優先

custom-prompt-library.json を失わない移行を先に入れ、タグスキーマv2を作る。 canonical名、翻訳、alias、カテゴリ、positive/negative、出典、モデル適性、使用履歴を分けて持つ。

UI方針

タグ追加は常時小さい「+」だけにせず、グループヘッダーに明確な追加ボタン、検索からの即追加、複数行貼り付け、候補付きtoken inputを併用する。

注意点

Tag-Managerにはデータ消失とAIダウンロード修正の未マージPRが残る。Yoitomoshiでは保存前バックアップ、dry-run import、migration testを必須にする。

2026-05-22時点の外部ベストプラクティス

分野調査結果採用判断
タグ入力UI WAI-ARIA Comboboxは、入力欄と候補popup、Enter/Escape/Arrowの操作を明確に定義している。Primerは「値がリストである入力」にはtoken inputを使い、Autocompleteと組み合わせる想定を示している。 タグ追加フォームは、自由入力 + 候補選択 + token削除を同じ部品として設計する。見た目だけの小ボタンではなく、キーボードで完結できる追加導線にする。
大量候補 MUI Autocompleteはfree solo、creatable、multiple values、virtualization、listbox roleなどを明文化している。 YoitomoshiではMUI導入は不要だが、設計原則は使う。候補数が増えるため、検索・virtual list・候補の「Add ...」行を想定する。
タグ知識ベース Danbooru系はtag category、alias、implication、related tagsを分ける。aliasはcanonical化、implicationは関連タグ自動追加に使える。 alias/implicationをYoitomoshi内のローカル知識として持つ。外部booruに書き戻すのではなく、prompt assemblyの補助として使う。
Civitai 公開APIは2026年4月にdeveloper portalへ移動した案内があり、現行の /api/v1/tags はライブ確認でタグ名とモデル検索リンクを返した。 Civitaiタグは「モデル検索・分類用」として保存し、生成プロンプトタグとは別フィールドにする。API失敗時のcache/fallbackを前提にする。
画像tagger PixAI Tagger ONNXは13,461タグ、general/character別thresholdを提示。WD EVA02 v3はrating/character/general tagを扱い、Danbooru画像で学習している。 tagger出力はscore/threshold/model/sourceを付けて「候補」として保存する。自動でライブラリに確定追加しない。

推奨するタグスキーマv2

現在の { en, ja } は表示には軽い一方、今後のLoRA/モデル別自動整形・tagger候補・表記ゆれ統合には不足します。 既存データを失わないため、v1を読み込んでv2へ変換し、保存はv2、UIは互換表示にします。 

{
  "schemaVersion": 2,
  "categories": [
    {
      "id": "people",
      "label": { "ja": "人物", "en": "People" },
      "groups": [
        {
          "id": "character-count",
          "label": { "ja": "キャラクター", "en": "Character count" },
          "tags": [
            {
              "id": "tag_1girl",
              "canonical": "1girl",
              "aliases": ["one girl", "girl solo"],
              "label": { "ja": "1人の女の子" },
              "polarity": "positive",
              "type": "danbooru-general",
              "modelFamilies": ["sd15", "sdxl", "illustrious", "noobai", "animagine"],
              "source": [
                { "kind": "built-in", "confidence": 1.0 },
                { "kind": "tagger", "model": "wd-eva02-large-tagger-v3", "confidence": 0.93 }
              ],
              "usage": { "count": 0, "lastUsedAt": null }
            }
          ]
        }
      ]
    }
  ]
}
フィールド目的最初の実装範囲
canonical保存・比較・重複排除の基準名。必須。空白/underscore正規化ルールも固定する。
aliases表記ゆれ、旧名、和製入力の吸収。手動編集 + import時の重複検出から開始。
polaritypositive/negative/bothを分離。既存negative欄移動と連動。
modelFamiliesPony/Illustrious/SDXL/SD1.5などで使える語彙を分ける。まず手動タグ付け。LoRA/Checkpoint profileとの連携は次段階。
sourceCivitai、tagger、built-in、手動、importの由来を保持。保存時に最低 manual / built-in を入れる。
usage最近/頻度/おすすめ順位に使う。既存recentTagsからv2へ吸収。

UI強化案

タグ追加ボタン周り

  1. グループ見出し右側に「+ タグ」を常時表示し、hoverしなくても押せるようにする。
  2. タグ欄の末尾にある小さい + は補助導線に下げ、主導線は見出しへ移す。
  3. クリック時はインライン1行入力ではなく、横幅のある追加パネルを開く。入力欄、翻訳、positive/negative、alias、モデル適性、出典メモをまとめる。
  4. 検索欄で該当タグがない場合、候補リストの最上部に Add "入力文字" を出す。
  5. 複数行貼り付けは残し、previewで「追加」「重複skip」「alias候補」を事前表示する。

タグ管理タブの情報設計

エリア役割追加したい操作
左ペイン カテゴリ、グループ、お気に入り、最近、未整理。 未分類、重複候補、alias未確認、tagger候補のフィルタを追加。
中央ペイン タグ一覧と検索。 token input、bulk paste preview、候補score、モデル適性バッジ。
右ペイン 選択タグ詳細。 canonical/alias/translation/source/modelFamilies/negative移動/削除履歴。
下部 現在のprompt/negativeへの反映。 選択タグをpositive/negativeへ移動、implicationを含めて追加、モデル別整形。

段階的ロードマップ

Phase実装内容検証リスク
1. 安全な土台 保存前バックアップ、atomic write、export/import、dry-run preview、v1 fixtureを用意。 既存 custom-prompt-library.json を読み込み、保存後もタグ数・カテゴリ数が一致するテスト。 ここを飛ばすとTag-Managerと同じデータ消失リスクが残る。
2. UI改善 グループ見出しの常時追加ボタン、追加パネル、検索からAdd、bulk preview。 DOM QAで data-testid による追加導線、キーボード操作、重複skipを確認。 見た目変更だけで終わると、データ品質は改善しない。
3. スキーマv2 canonical/alias/source/modelFamilies/polarity/usageを導入し、既存UIには互換アダプタを通す。 migration unit test、typecheck、既存PromptLibrary表示のsnapshot相当確認。 一括変換で翻訳やお気に入りを落とさないようにする。
4. 推薦・連携 Civitai/LoRA profile/tagger/history reviewから候補を出し、ユーザー承認後にライブラリへ昇格。 tagger score付き候補を保存、採用/拒否の履歴が次回候補順位に反映されること。 自動追加しすぎるとライブラリが汚れる。必ずreviewを挟む。
5. モデル別プロンプト整形 Pony/Illustrious/NoobAI/Animagine/SD1.5などのprofileに基づき、quality tag、negative、tag順序を調整。 Checkpoint profileとLoRA補助入力の組み合わせで、重複や衝突が出ないこと。 モデル固有ルールは時間で変わるため、手動編集可能にして固定しすぎない。

2026-05-22 実装結果

Phase 1〜3 の中核を実装済み。タグ追加UI、schema v2、保存前バックアップ、atomic write、 JSON import dry-run preview、実ファイルを使ったDOM QAまで通過した。

  • src/components/PromptLibrary.tsx: タグ追加パネル、alias/canonical対応、取込プレビュー、適用時マージを追加。
  • src/shared/types.ts: PromptLibraryDocumentV2 とタグメタデータ型を追加。
  • electron/storage.ts: legacy配列とschema v2の両対応、保存前バックアップ、atomic writeを追加。
  • scripts/dom-qa.cjs: tag-library-import QAを追加。schema v2とlegacy配列の一時JSONを選択し、新規/重複/無効件数、適用後保存、cleanupを確認。

実装時の受け入れ条件

  • 既存のbuilt-in YAML、user custom、favorites、recentTagsを読み込める。
  • 保存前に userdata/backups/tag-library/ へ自動バックアップが残る。
  • importはdry-runで追加数、skip数、重複、alias候補、破損行を表示する。
  • タグ追加UIはクリック操作だけでなく、Enter/Escape/Arrow/Backspaceの最低限のキーボード操作を持つ。
  • 候補は自動確定せず、Civitai/tagger/historyなど出典を表示して承認後に保存する。
  • 検証は npm.cmd run typecheck、関連DOM QA、migration unit test、git diff --check を通す。

採用しない方針

  • Tag-ManagerのTkinter UIをそのまま移植しない。YoitomoshiのReact/Electron UIと操作体系が分断されるため。
  • SQLiteへ急に置き換えない。現状のYAML/JSON資産とElectron storageに合わせ、まずschema versioningで進める。
  • AI/tagger結果を自動で全追加しない。画像taggerは便利だが、誤検出やモデル由来の偏りがあるためreviewを挟む。
  • Civitaiタグをそのままpromptタグ扱いしない。Civitaiのタグはモデル分類用途が混ざるため、source/typeを分ける。
