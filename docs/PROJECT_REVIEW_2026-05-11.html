コードレビュー所見

P1 Electron userData をプロジェクトルートにしている

electron/main.ts:31 で app.setPath('userData', projectRoot) を設定しているため、 Cache、Code Cache、GPUCache、Local Storage、Network、Preferences などがルート直下に生成されています。 ポータブル方針自体は正しいですが、Electronのプロファイルとアプリのユーザーデータを同じ階層に置くのは事故りやすいです。

改善: Electron profile は userdata/electron-profile 等へ隔離し、Storageのrootは userdata を明示的に使う。合わせて既存のroot直下キャッシュを移動または削除対象にする。

P1 Civitai APIキーの保存方式

初回レビュー時点では userdata/settings.json にAPIキーが入っていました。値はこのレポートには記載しません。 2026-05-11の整理で settings.json から secrets.local.json へ分離し、設定ファイル上は civitaiApiKey: null になりました。

残リスク: 初回移行値は plain: base64 fallback です。Electron safeStorage が使える環境では次回保存時に暗号化形式へ更新されます。userdata/ は引き続きGit禁止です。

P1 IPC入力検証が機能ごとに薄い

初回レビュー時点では toolsInspectModel、storageSetSettings、openExternal、civitaiDownload の入力検証が薄い状態でした。 2026-05-11の境界強化で、モデル検査はユーザー選択済みファイルまたはForgeモデル配下、外部URLは https + 許可ドメイン、設定保存はForge実体と型検証、Civitai downloadはURL/filename/hash検証へ変更済みです。 txt2img / img2img / extras / interrogate payloadも、文字列長、画像base64サイズ、steps / cfg / size / seed / batch / denoise などを検証してからForgeへ渡すようにしました。

残リスク: manifest normalizeは追加済みです。次は設定変更時のUI側警告、payload契約テスト、既存モデルへのmetadata再取得が残っています。

P2 Electron security は基本は良いが詰めが残る

nodeIntegration: false、contextIsolation: true は良い設定です。一方、electron/main.ts:97 で sandbox: false、 ipc-handlers.ts:822 で任意URLを shell.openExternal に渡せる設計です。外部ページを直接読み込んではいないため緊急度はP1より下ですが、Electron公式のセキュリティ推奨に照らすと改善余地があります。

改善: will-navigate を拒否、外部リンクは https: のみかつ civitai.com などのallowlistへ限定、renderer sandboxを有効化できるか検証する。

P2 自動テスト不足に対して中核payloadが複雑

src/lib/extension-payload.ts はADetailer、ControlNet、MultiDiffusion、Ultimate SD Upscaleのwire formatを集中管理しており、壊れると生成全体が失敗します。 コメントで経緯はよく残っていますが、positional argsはGitHub側の拡張更新で壊れやすい領域です。

改善: まずVitest等で buildAlwaysOnScripts() のスナップショット/契約テストを作る。ADetailer最小dict、ControlNet unit、Upscale Diffusion、Ultimate script_args 18要素を固定化する。

P2 大きいファイルが分割時期に来ている

主な大物は src/lib/i18n.ts 2391行、electron/civitai-api.ts 1024行、src/components/UpscaleWorkspace.tsx 959行、 src/components/PromptLibrary.tsx 873行、electron/ipc-handlers.ts 778行です。今は動いていますが、次にToolsやInpaintを足すと変更衝突とレビュー負荷が上がります。

改善: 機能単位で electron/ipc/*、src/features/upscale/*、src/i18n/*.ts へ分割する。分割はテスト追加と同時に小さく行う。

P3 ビルド警告とGit前提の弱さ

postcss.config.js のmodule type警告、png-metadata.ts のdynamic/static import警告は動作阻害ではありません。 ただしローカルがGitリポジトリでないため、差分管理、GitHub Issues/PR、リリースタグが使えません。

改善: git init 前に .gitignore を拡張し、runtime/、Electron profile、cache類、userdata/、生成物を除外する。

強み

  • ForgeをGradio UIではなくAPI専用で扱う方向は正しい。Electron UIを主導権にでき、個人制作向けの導線を作りやすい。
  • Forge終了時のプロセスツリー停止、stale pid cleanup、API readiness polling は実運用上の問題に寄り添っている。
  • Civitai hash lookup、community mining、LoRA/VAE推奨、履歴再利用は「素材制作の反復」に直結している。
  • ControlNet Tileを通常ControlNetと分離し、Upscale専用payloadにしている点は副作用が少ない。
  • safetensors ヘッダのみを読んでModel Health Scanする設計は、大容量モデルを扱うアプリとして効率が良い。
