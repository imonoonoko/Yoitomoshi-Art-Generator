Findings First

P1

作業ツリーが大きくdirtyで、レビュー時点の実装単位がまだ固まっていない

git status --short --branch は modified 23件、untracked 39件、合計62件を示した。 Prompt大辞典、source registry、importer、DOM QA、docs、DB生成物が混在している。現在の機能は動くが、 後から復元・比較・切り戻しするには変更単位を分けてコミットする必要がある。

推奨: まず「Prompt大辞典source/ingest」「Autocomplete/Dictionary UI」「docs/report」「生成物DB」の4単位に分けて差分を確認する。

2026-05-29追記: GitHub公開用のDBビルドでは、ローカル履歴由来の local-user-prompts を既定で除外するよう整理した。

P1

今回レビューでは実生成のForge core smokeを走らせていない

npm.cmd run typecheck、npm.cmd run build、Selectors、API surface、Prompt大辞典DOM QAはPASS。 ただし、画像生成そのものを行う qa:dom:forge-core / qa:dom:forge-controlnet は今回未実行。 Generate payloadやForge拡張を触る変更ではないためレビューには十分だが、生成品質保証としては別枠で実行が必要。

推奨: 生成系を触る前後は qa:dom:forge-core -- --port=9338 を直列で走らせる。

P2

Forge ready時間は改善済みだが、まだばらつきが大きい

直近50 startup metricsでは Forge ready 平均 23.7秒（23,684ms）、最小 14.8秒、最大 56.7秒。 renderer loaded平均は445msなので、体感起動の主コストは依然としてForge/Python/Torch/extension側。

推奨: 遅いサンプル発生時に Tools > 個人環境ヘルスチェック とForge log tailを同時に保存する。

P2

履歴・Pro Recipe・Prompt Composerの「知識化」機能は実装に対してデータ利用が薄い

userdata/history/index.json は500件あるが、現在の集計では tagReview 0件、proRecipeReview 0件、ラベル付きはfavorite 1件のみ。 checkpoint-prompt-profiles.json は22件、lora-usage.json は1,061件あり、モデル/LoRA側の蓄積は進んでいる。

推奨: まず直近50件だけに採用理由・失敗理由・用途ラベルを入れる導線を短くする。機能追加よりデータ化の摩擦削減が効く。

P2

Model Libraryは健全だが、重いモデル在庫とunsupported隔離は定期レビュー対象

主要モデルカテゴリ合計は約62.3GB。Model Library indexは45件、missing file 0、SHA既知45件、Civitai/source metadata 43件、preview 38件。 Unsupported-StableDiffusion に1件約3.89GBが残っており、通常Forge生成に使わない資産は削除ではなく候補レビュー扱いが妥当。

推奨: 削除は明示指定がある時だけ。通常は「unsupported / Comfy-only候補」としてレポートに残す。

P3

ドキュメントは豊富だが、入口の鮮度と重複が次の保守ポイント

docsは93ファイル。DOCS_INDEX.md は2026-05-27更新済みだが、AI_PROJECT_MAP.md と ROADMAP.md の先頭日付は2026-05-25。 直近のPrompt大辞典とAutocomplete実装を、入口ドキュメントにも反映すると次回の探索が速くなる。

現在の結論

このcheckoutは、自分用制作環境としては十分に実用域。 生成UI、Prompt資産、辞書、モデル管理、履歴、Upscale、Video、保守診断が一体化しており、単なるForge launcherではない。 ただし「外部配布」「第三者が迷わず再現」「クリーンなリリース」には、dirty差分整理、生成core QA、docs入口更新が必要。
