PixAIの一次情報から二次情報までを調査し、Yoitomoshi Art Generatorへ吸収できるワークフロー、UI、機能設計、リスクを整理した実装判断用レポート。

1. 結論

最も吸収価値が高い要素

PixAIの強みは、個別機能の数よりも「生成前、生成中、生成後、再編集、モデル探索、学習、共有」が一つの創作ループとしてつながっている点にある。

結果アクション 設定テンプレート 自然文補助 Market連携 ControlNet導線

Yoitomoshiで先にやるべきこと

PixAIのクラウド機能を再現するより、ローカルForgeの強みを活かし、生成結果カードから次の操作へ進む導線を整えるのが最短で効果が出る。

Variation Use as Base Image Enhance/Upscale Recipe保存

コピーしないもの

PixAIのUIデザイン、商用導線、クレジット制、独自モデル、コミュニティ運用はコピー対象ではない。採用するのは概念とユーザー導線だけ。

UI資産コピー禁止 商用導線は対象外 API依存は任意機能扱い

3. PixAIの機能構造

3.1 生成パネル

PixAI公式ヘルプでは、プロンプトはタグ/短文をカンマ区切りで書くのが基本とされ、Autocomplete、強調記法、Note、Prompt Helper、Image to Promptが生成前の補助として並ぶ。Noteはプロンプト、ネガティブ、モデル、サイズ、sampling steps/methods、base imageをテンプレートとして保存する機能と説明されている。

Yoitomoshiへの示唆: 現在のWorkspace保存を「生成レシピ」として前面に出し、テンプレート名、説明、タグ、サムネイル、モデル互換性、再現性チェックを持たせる。

3.2 自然文入力とPrompt Helper

PixAIは自然文で書いた内容をPrompt Helperで生成向けに変換する導線を持ち、多言語対応を打ち出している。Platform API v2にも promptHelper パラメータがあり、生成時に自動補助を有効/無効化する設計になっている。

Yoitomoshiへの示唆: 外部AI APIを前提にせず、まずは既存Prompt Library、履歴、Civitai metadata、LoRA trigger wordsからローカル補助を作る。将来、任意の外部LLM連携を追加できる余地を残す。

3.3 生成結果アクション

PixAIの生成結果には、Animate、Describe、Delete、Variation、Edit、Use as Base Image、Download、Enhance、Publish、Favoriteが並ぶ。特にVariationは、現在画像をbase imageにし、同じprompt/settingsから3枚の別案を出すワークフローとして説明されている。

Yoitomoshiへの示唆: 生成結果カードの主要アクションを、アイコンだけでなく用途別に整理する。既存の「img2imgへ送る」「Upscaleへ送る」「Variation比較」を、PixAI型の結果後ワークフローとして統合する。

3.4 編集、Inpaint、Outpaint、Enhance

PixAIは参照画像、Paint、Image Edit、Inpaint、Outpaint、Enhanceを生成パネルからつなげている。Edit Proの記事では、自然言語指示、複数画像入力、構図維持、キャラクター一貫性が強調されている。

Yoitomoshiへの示唆: いきなり高機能AI編集を作るより、既存img2img/Inpaint/Tile Upscaleを「選択範囲を直す」「背景を足す」「高解像度化する」の3つの明確な導線へ整理する。

3.5 Model / LoRA Market

PixAIのModel Marketは、キーワード、タグ、Base Model、Source、Category、Submission Timeで絞り込み、モデル/LoRA詳細から「Use this model/LoRA」で生成ページへつなぐ。選んだモデルは生成設定のデフォルトになり、LoRAはタスクに追加される。

Yoitomoshiへの示唆: Civitai/Hugging Face検索結果に「このモデルで生成」「LoRAを追加」「互換base model警告」「お気に入りモデルリスト」を追加する。ファイルを落とすだけでなく、生成状態へ直接つなげる。

3.6 LoRA Training

PixAI公式のLoRA Training Guideは、解像度、明瞭な特徴、ウォーターマーク除去、サイズ正規化、データセット枚数、base model互換、trigger wordsの設計を重視している。特に「base model compatibility」と「trigger wordsは恒久特徴に絞る」という考え方はローカル運用にも有効。

Yoitomoshiへの示唆: LoRA学習そのものをすぐ実装しなくても、LoRA管理画面に「base model」「trigger words」「推奨weight」「用途: character/style/pose/outfit/detail」を持たせるだけで運用品質が上がる。

3.7 ControlNet / Composition

PixAI公式ヘルプではControlNetを、アップロード画像からフレームや形を得て生成へ反映する機能として説明している。Compositionはキャンバスの各領域に何が現れるかを指定する機能だが、SDXLモデルでは利用不可と注記されている。XLモデルの公式ヘルプでも、通常LoRAとXL LoRAの非互換、ControlNet/Composition/Outpaintingの一部非対応が説明されている。

Yoitomoshiへの示唆: ControlNetを「技術パネル」ではなく、ポーズ、線画、Depth、Tile、Referenceといった役割別カードへ変換する。SDXL/SD1.5互換性はUIで明示する。

3.8 API / Platform

PixAI PlatformはREST API、API key、task status、webhookを提供する。v2の画像生成APIは、modelVersionId、prompt、negativePrompt、aspectRatio、mode、style、loras、seed、promptHelper、callbackUrlなどを扱う。

Yoitomoshiへの示唆: PixAI APIを必須化しない。ローカルForgeを主軸にしつつ、将来の任意クラウドバックエンド抽象化の参考に留める。

4. 二次情報から見える摩擦点

観測 根拠 Yoitomoshiでの対応
機能が多いほど、モデル互換・ControlNet互換・SDXL制約が混乱源になる。 第三者レビューはXL制約、Composition/ControlNetの互換性、モバイル機能差を摩擦点として挙げている。 モデル選択時に「このLoRA/ControlNetは互換」「不明」「非互換」を表示し、失敗前に止める。
クレジット制は便利さとコスト不透明感が同居する。 PixAI MembershipはTurbo、Video、Flow Edit、LoRA training、追加creditを売りにする一方、第三者レビューは行動ごとの費用把握の難しさを指摘。 ローカルアプリでは「VRAM/所要時間/ディスク/モデルサイズ」をコストとして見える化する。クラウド費用ではなくPC負荷の透明化が価値になる。
モバイルや初心者には「ファイルを移動しなくてよい」ことが大きい。 Google Play掲載はModel Market、編集、LoRA training、ControlNet、Hi-Res Upscaling、日次credit、コミュニティを訴求している。 YoitomoshiはModel LibraryとDownload Managerを強化し、Civitai/HFから正しいForgeフォルダへ自動配置する価値を前面に出す。
ユーザーは高品質化に敏感だが、アップスケール/Enhanceのコストや失敗も気にする。 PixAI公式とレビューではEnhance/HiRes/Videoが重要機能として扱われ、ユーザーコメントには高画質化のcredit負担への言及もある。 Upscaleはdenoise比較、Tile ControlNet、低負荷設定、プレビュー比較を明確に出す。

5. Yoitomoshiへ吸収する実装候補

優先 候補 内容 実装難度 理由
P1 生成結果カード再設計 Variation / Use as Base Image / Enhance / Edit / Recipe Save / Download / Favoriteを同じカード上に整理。 中 既存機能をつなぎ直すだけで、PixAI的な創作ループに近づく。
P1 生成レシピ強化 .yoitoartをNote相当に育て、設定テンプレート、サムネイル、タグ、モデル互換性、入力画像参照を保存。 中 再現性と初心者導線の両方に効く。
P1 Marketから生成へ直結 Civitai/HF検索結果に「このモデルで生成」「LoRAを追加」「お気に入り」「base model警告」を追加。 中 PixAIのMarket体験に最も近い。既存Civitai基盤を活かせる。
P2 ローカルPrompt Helper 自然文をタグ候補、ネガティブ候補、LoRA候補、ControlNet候補へ分解。外部APIなしから開始。 中から大 PixAIの初心者向け強みをローカルで再現できる。
P2 ControlNet Builder ポーズ/線画/depth/tile/referenceの役割カードと互換性警告を追加。 中 既存ControlNetパネルを初心者に見える形へ変換する。
P2 LoRA metadata管理 base model、trigger words、用途、推奨weight、互換性、サムネイルをModel Libraryへ持たせる。 中 LoRA Trainingそのものより先に、運用管理の質を上げられる。
P3 Image-to-Video連携 ローカル動画生成または外部ツール連携の研究項目として扱う。 大 PixAIの差別化要素だが、ローカルForge本体からは距離がある。

6. 直近スプリント提案

  1. PreviewPanel結果アクション整理: 既存のimg2img直送、Upscale直送、保存、履歴、Variationを1つの結果アクション列として再配置する。
  2. Recipe Saveボタン: 現在画像と現在設定をまとめてWorkspace/Recipeへ保存するボタンを生成結果カードに追加する。
  3. Use as Base Imageの明文化: ボタン名とツールチップを明確にし、img2img入力へ送った後のタブ遷移を自然にする。
  4. Enhance/Upscale導線: 生成結果からUpscaleタブへ送る際、Tile ControlNet推奨、denoise比較、倍率プリセットを同時に提案する。
  5. Favorite/Statusラベル: 履歴に「お気に入り」「採用候補」「没」「素材化済み」を付ける最小実装を追加する。
判断: PixAIの研究から、次の実装は「新しい巨大機能」よりも、既にある生成・履歴・img2img・Upscale・Workspaceを結果カード中心につなぎ直す方が効果的。

7. リスクと注意点

  • ポリシー: PixAI公式ガイドラインはNormal/Sensitive分類、AI moderation、未成年関連の厳格な禁止、公開制限を持つ。Yoitomoshiはローカルアプリでも、少なくとも外部投稿やモデル配布機能を追加する時はルール設計が必要。
  • 互換性: SDXL、XL LoRA、通常LoRA、ControlNet、Compositionの互換性は混乱しやすい。PixAIも制限を明記しているため、Yoitomoshiは事前警告を強くする。
  • API依存: PixAI Platform APIは便利だが、Yoitomoshiの中核はローカルForge。クラウドAPI連携は任意機能に留める。
  • UIコピー: PixAIの画面や文言をコピーせず、概念をYoitomoshiの既存UIに合わせて再設計する。
