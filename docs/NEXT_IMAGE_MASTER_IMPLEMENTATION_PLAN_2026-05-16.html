Master Implementation Plan

次の一枚: 生成の試行錯誤を勝手に整理する制作台

この計画書は、Romptn横断リサーチ、ローカル生成履歴、既存のPreview/Variation/History/Prompt Helper実装を前提に、 「プロンプトをうまく書かせるソフト」ではなく「当たり画像までの試行錯誤を短くするソフト」へ寄せるための実装計画である。 100回の設計反復に相当する観点を、実装順・UI仕様・データ設計・検証条件へ圧縮した。

作成日: 2026-05-16 対象: Yoitomoshi Art Generator 主戦場: Preview / History / Variation 方針: 辞書を見せず、比較を前へ

一文で決める

生成結果を見た直後に、ソフトが「次に試すべき一枚」を提示し、似た試行を自動で束ね、比較単位で当たりを探せるようにする。

これはPrompt Helperの拡張ではない。Previewの下に置く「次の一手」機能であり、Historyを比較ボードへ変えるための入口である。

履歴が示す使い方

  • 同じ方向性のpromptを、seed、steps、cfg、model、LoRA有無で連続的に試している。
  • ラベルは全履歴で未使用。つまり「後で分類する」より「今見て選ぶ」方が自然。
  • 縦長キャラクター生成が中心。結果画像と次アクションが主役になる。
  • 特定モデルに集中しているため、モデル切替は日常操作ではなく比較アクションとして扱うべき。

既存実装が示す近道

  • PreviewPanel には既に Variation、img2img送信、Upscale送信、Recipe保存の出口がある。
  • VariationPanel はseed/prompt/cfg/denoise比較生成の土台を持っている。
  • PromptHelperPanel はローカル辞書候補をappendする形で安全に動く。
  • HistoryGallery はラベル、復元、比較の接続点を持つ。ここを「倉庫」から「比較ボード」へ寄せる。

機能仕様

次の一枚パネル

Preview下部に出る主UI。生成結果がない時は非表示。生成中は読み取り専用または無効。

  • この方向で続ける: seedだけ変えて継続。
  • seed 4枚: 既存Variationのseed軸を4候補で実行。
  • 表情だけ: 表情/視線タグのみ差し替え候補を提示。
  • 構図だけ: shot/angle/pose系タグだけ差し替え候補を提示。
  • 背景だけ: location/time/weather/light系タグだけ差し替え候補を提示。
  • 失敗を減らす: negativeに目的別の短いセットを追加。
比較レーン

VariationPanelを「実験候補」表示として再利用する。最初は新しい生成キューを作らない。

  • seed比較: 既存実装をそのまま利用。
  • prompt比較: dynamic promptまたは候補差し替えを使う。
  • model比較: 後段。選択済みモデル周辺から2候補だけ。
  • LoRA比較: 後段。現在のLoRAあり/なし、weight 2段階。
履歴の自動まとまり

Historyを「倉庫」から「同じ実験の比較ボード」へ変える。永続化は安定後。

  • prompt fingerprintで似た履歴を束ねる。
  • seed、qualityタグ、weight表記、LoRA tokenはfingerprintから除外。
  • model、LoRA有無、サイズは差分軸として見せる。
  • 手動ラベルは残すが、最初に要求しない。
Romptnの扱い

Romptn記事のタグ群は辞書画面に大量表示しない。小さなaction presetへ圧縮する。

  • 表情: smile / grin / serious / displeased / blush など短いセット。
  • 視線: looking at viewer / looking away / looking back。
  • 構図: full body / cowboy shot / from below / from behind。
  • 背景: outdoors / indoors / night / city street / sky。
  • negative: 手、顔、ぼけ、文字、余計な四肢を目的別に。

型の最小案

export type NextImageActionId =
  | 'continue-seed'
  | 'seed-compare'
  | 'expression-variant'
  | 'composition-variant'
  | 'background-variant'
  | 'reduce-failure'

export interface NextImageAction {
  id: NextImageActionId
  labelKey: string
  summaryKey: string
  icon: 'shuffle' | 'sparkles' | 'camera' | 'image' | 'shield'
  kind: 'patch' | 'variation'
  patch?: {
    prompt?: string
    negativePrompt?: string
    params?: Partial<GenerationParams>
  }
  preview: {
    addedPositive: string[]
    removedPositive: string[]
    addedNegative: string[]
    paramChanges: string[]
  }
}

fingerprintの最小案

normalizePromptForFingerprint(prompt):
  remove adapter tokens: <lora:...>, <lyco:...>
  remove weights: (tag:1.2) -> tag
  remove common quality tags: masterpiece, best quality, high quality
  remove seed-only/dynamic noise when detectable
  normalize spaces, underscores, duplicate commas
  keep subject, outfit, composition, background, key style tags
  sort stable low-impact tags, preserve strong subject tokens
  return hash(normalized tokens)

実装フェーズ

P0

計画固定と安全境界

このHTML計画書を基準に、最初の実装範囲をPhase 1だけへ固定する。

  • Preview直下に出す。PromptPanel主導にしない。
  • 永続化、生成キュー、History再設計は入れない。
  • 既存変更が多い作業ツリーなので、差分は小さく保つ。
完了条件: HTML計画書、実装対象ファイル、禁止事項が明確。
P1

次の一枚パネル MVP

生成結果の下に、6つのアクションを表示し、押すと変更案を適用できる状態にする。

  • next-image-actions.ts を追加。
  • NextImagePanel.tsx を追加。
  • PreviewPanel.tsx へ接続。
  • 変更前に追加/削除/params変更を表示。
  • seed比較は既存VariationPanelのseed軸へ誘導。
完了条件: 生成後だけ表示、各ボタンでprompt/negative/paramsが期待通り変化。
P2

候補の質を上げる

Romptn由来の候補を、大量辞書ではなくaction presetへ圧縮する。

  • 表情、視線、構図、背景、照明、negativeの短いセットを追加。
  • 既存promptに同系統タグがある場合は、追加ではなく差し替え候補にする。
  • モデル依存の候補には注意文を出す。
  • ユーザーがよく使う履歴トークンを候補順位に軽く反映する。
完了条件: 候補が3〜6個に収まり、タグ一覧UIにならない。
P3

Historyを同じ実験で束ねる

表示時fingerprintで似た履歴をグループ化し、同じprompt周辺を横並びで見せる。

  • history-fingerprint.ts を追加。
  • HistoryGalleryに「同じ実験」フィルタまたはセクションを追加。
  • seed違い、model違い、LoRA違いをバッジで表示。
  • 手動ラベルは既存のまま残す。
完了条件: 181件履歴で重くならず、同じ実験が自動でまとまる。
P4

ワンクリック比較生成

既存VariationPanelを発展させ、比較軸をユーザーの作業に近い言葉へ変える。

  • seed 4枚 を最初の標準比較にする。
  • 表情3種、構図3種、背景3種 を追加。
  • LoRAあり/なし はactiveLorasがある時だけ表示。
  • モデルA/B はモデル候補の安全な選択仕様を固めてから。
完了条件: 比較候補がHistoryへ残り、採用画像をimg2img/Upscaleへ送れる。
P5

採用判断の流れを作る

当たり候補を選んだ後の行き先を短くする。ここで初めてラベルやWorkspaceを自然に使わせる。

  • 比較候補から「採用」「もう一回」「Upscale」「img2img」「Recipe保存」を出す。
  • 採用時だけHistory labelのcandidate/favoriteを自動付与する案を検討。
  • Workspace保存は明示操作のまま。勝手に巨大ファイルを作らない。
完了条件: 当たり画像から次工程へ迷わず進める。

表示ルール

  • lastImage === null: 非表示。
  • isGenerating === true: 表示してもdisabled。
  • activeLorasなし: LoRA比較は出さない。
  • img2img入力なし: denoise比較は出さない。
  • promptが空: 表情/構図/背景は「候補追加」だけにする。

ボタン文言

この方向seedを変えて近い方向をもう一枚。
seed 4枚同条件で4候補を見る。
表情表情・視線だけを小さく変える。
構図距離・角度・向きだけを変える。
背景場所・時間帯・光だけを変える。
失敗減らすnegativeを補強する。

  • Romptn記事名やタグ一覧。
  • prompt fingerprintの内部値。
  • 難しい分類カテゴリ。
  • モデル依存の断定表現。
  • 外部API連携や投稿サイト連携。

採用しない案

巨大Prompt Library化

候補が多いほどよい、という設計にはしない。今回の価値は一覧ではなく、今の画像に対して次に試す候補を絞ること。

Prompt All in One直接導入

API専用Forge起動ではGradio UI依存が邪魔になりやすい。既存方針通り、発想だけReact側に取り込む。

最初から自動生成キュー

便利だが失敗時の責務が大きい。P1ではstate patchと既存Variationの呼び出しに留める。

履歴グループの即時永続化

fingerprint仕様が固まる前に保存形式を増やすと後戻りが重い。まず表示時計算にする。

検証計画

  • typecheck: npm.cmd run typecheck
  • build: npm.cmd run build
  • diff: docsや辞書変更時は git diff --check
  • DOM smoke: Electron remote debugging + data-testid で確認。
  • 表示条件: 生成結果なしでは非表示、生成結果ありで表示、生成中はdisabled。
  • action: 各ボタンがprompt/negative/paramsの狙った場所だけ変える。
  • 重複: 同じnegativeや同じ表情タグを増やし続けない。
  • 既存機能: Variation、img2img送信、Upscale送信、Recipe保存が壊れない。
  • 多言語: 日本語/英語/ロシア語/ポルトガル語でセレクタが文言依存しない。
  • 履歴: P3時点では181件程度の履歴で表示が重くならない。
