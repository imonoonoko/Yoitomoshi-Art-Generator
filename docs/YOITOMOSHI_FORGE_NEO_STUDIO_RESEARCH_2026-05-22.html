Yoitomoshi Forge Neo Studio 制作リサーチ

現行Yoitomoshi Forge Studioを土台に、Forge Neoを別runtimeとして採用する新アプリ 「Yoitomoshi Forge Neo Studio」を作る案の実現性、リスク、移行方針、最小実装範囲を調査した。 Anima対応だけでなく、Qwen-Image、Wan 2.2、Flux.2-Kleinなど新世代モデルGUIへの発展余地も含めて評価する。

結論: Yoitomoshi Forge Neo Studioを新規派生アプリとして作る案は現実的。 ただし現行 runtime/forge をNeoへ上書きするのではなく、runtime/forge-neo などの別runtime、 別ポート、別設定で段階導入するべき。最大の価値は「ComfyUIほどノードを触らず、Anima / Qwen / Wan系をWebUI風に使える」 こと。最大のリスクはForge Neo側の更新速度、API差分、拡張互換、AGPL-3.0配布対応。

1. 何を作るべきか

案 評価 理由 推奨
現行Yoitomoshi Forge StudioをNeoへ全面置換 非推奨 現行アプリは同梱Forgeの system/python/python.exe と webui/launch.py を直接spawnし、API専用 --nowebui 運用に最適化されている。 NeoはPython 3.13 / uv / venv前提の導入情報があり、起動、依存、API、拡張の差分が大きい。 既存の安定Forge経路を残す。
Yoitomoshi Art Generator内にNeo engineを追加 中期なら可 UI部品と履歴、モデルライブラリを共有できる一方、設定、runtime、生成payload、モデル選択UIが複雑になる。 既存ユーザーの安定性を守るにはfeature flagと移行ガードが必要。 最初の実験には少し重い。
Yoitomoshi Forge Neo Studioとして兄弟アプリ化 最推奨 現行Forge版を壊さず、Neo専用のruntime manager、モデルmodule UI、Anima preflightを素直に設計できる。 既存YoitomoshiのReact/Electron UX資産は継承しつつ、バックエンド前提を切り替えられる。 まずこの形でMVPを作る。
製品ポジション: 現行Forge Studioは「安定したSD/SDXL制作環境」、Comfy Studioは「workflowを固定して強力なエンジンを隠す環境」、 Forge Neo Studioは「新世代モデルをWebUI風GUIで素早く触る環境」として分けると衝突しにくい。

2. Forge Neo一次情報の要点

項目 調査結果 Yoitomoshiへの影響
プロジェクト状態 Haoming02/sd-webui-forge-classic のdefault branchは neo。GitHub API調査時点で stars 1,226、forks 143、open issues 10、 最新releaseは 2.24、公開日は 2026-05-15、直近pushは 2026-05-22。 活発だが安定固定ではない。Yoitomoshi側ではtagまたはcommit pinが必要。
対応モデル READMEはAnima、Flux.2-Klein、Ernie-Image、Z-Image、Wan 2.2、Qwen-Image / Qwen-Image-Edit、 Flux Kontext、Nunchaku、Lumina-Image-2.0、Chroma1-HD、MixedPrecisionなどを列挙している。 Animaだけのために作るより、「Neo系モデルのGUIハブ」として設計する価値がある。
Anima配置 Forge Neo WikiはAnima本体を models/Stable-diffusion、 qwen_3_06b_base.safetensors を models/text_encoder、 VAEを models/VAE に置く形で案内している。 現行ForgeでUnsupportedに隔離していたAnima系DiTを、Neo Studioでは正規checkpointとして扱える。
起動環境 READMEはuvを推奨し、uv venv venv --python 3.13 --seed と --uv flagを示す。従来方式ではPython 3.13.12が推奨されている。 現行ForgeManagerの system/python 直spawnとは合わない。 Neo専用start strategyが必要。
モデル共有 READMEは --model-ref、--forge-ref-a1111-home、 --forge-ref-comfy-home、--forge-ref-comfy-yaml を案内している。 巨大モデルを二重配置しない設計が可能。YoitomoshiのModel Libraryと相性がよい。
Issue policy READMEは「削除機能」「ユーザーエラー」「AMD GPU」「非公式モデル」「3rd-party extensions」「Stability Matrix起因」のIssueは無視されると明記している。 Yoitomoshi側で拡張互換や利用者サポートを肩代わりする前提が必要。
ライセンス GitHub APIではForge Neoのlicenseは AGPL-3.0。 LICENSEはAUTOMATIC1111、lllyasviel、Haoming02の著作権表記を含む。 NeoコードをElectron本体へ混ぜない。runtime同梱や改変配布をする場合はソース提供、著作権表示、ライセンス表示を分離して管理する。

3. 現行Yoitomoshiから継承できるもの

領域 継承可否 根拠 Neo化で必要な変更
Electron / React UI 高 txt2img / img2img / Video / Upscale / Models / Tools の構造は、Neoでも制作UIとして再利用できる。 Anima、Qwen、Wanなどのmodel family別に、必要moduleと推奨値を出すpreflightを追加する。
Forge REST client 中 /sdapi/v1/txt2img、/img2img、/sd-models、 /sd-modules、/options、/progress は基本流用できる可能性が高い。 Neo用に forge_additional_modules の複数選択、UI preset、Low Bits、model family検出を扱う。
ForgeManager 低から中 現行は system/python/python.exe と webui/launch.py が必須。 Neo標準導入は venv / webui-user.bat / uv 寄り。 RuntimeStartStrategy を作り、Classic portableとNeo venvを分離する。
Model Library 高 既に models/text_encoder、Unsupported-StableDiffusion、 safetensors header分類、Civitai/Hugging Face download基盤がある。 Text Encoderを一級assetとして扱い、Anima / Qwen / Wanの「本体+modules」セット管理を追加する。
生成payload 中 現行は override_settings.sd_model_checkpoint と sd_vae が中心。 Forge側には forge_additional_modules が存在し、処理中に複数moduleを読む。 selectedVae だけではなく selectedModules: string[] を持つ。
ControlNet / ADetailer / AnimateDiff 要検証 Forge Neoのopen issuesにはControlNetやADetailer絡みがあり、READMEも3rd-party extension問題を上流サポート外としている。 MVPではAnima txt2imgを優先し、拡張payloadは段階的に戻す。

4. 推奨アーキテクチャ

4.1 runtime分離

項目 現行Forge Studio Forge Neo Studio案
runtime path runtime/forge runtime/forge-neo
既定port 7860 7861 推奨。現行Forgeと同時検証できる。
起動 system/python/python.exe webui/launch.py venv/Scripts/python.exe launch.py または webui-user.bat。 まず手動起動でAPI互換を確認してから自動起動を作る。
モデル Forge配下へ配置 --model-ref や --forge-ref-comfy-home で共有を優先。 ただしAnima smokeでは最初はNeo配下へ最小3ファイルを置く。
ライセンス 現在のElectron appはMIT、Forge runtimeは外部/生成物扱い Neo runtimeはAGPL-3.0として明確に分離。公開配布時はsource offerとnoticeを同梱。

4.2 engine adapter

UIは現行のYoitomoshiを継承しつつ、生成実行層を次のように分ける。

層 責務 Neo Studioでの変更点
Renderer UI prompt、negative、size、sampler、LoRA、ControlNet、history、model library model family preset、module bundle、Low Bits、Anima preflightを追加。
Generation Planner store状態からREST payloadを作る sd_vae 単独ではなく、Neo用に forge_additional_modules 配列を作る。 Animaなら qwen_3_06b_base と qwen_image_vae を必須化。
ForgeNeoApi A1111互換APIとNeo差分APIを包む /sdapi/v1/sd-modules、/options、/scripts、 /samplers、/schedulers のresponse差分を実測して型を分ける。
RuntimeManager 起動、停止、ready検知、ログ、プロセス掃除 ClassicとNeoを別strategyにする。Neoではuv/venv、FFmpeg、Sage/Flash/Nunchaku等のoptional dependencyを診断する。
Model Registry ローカルモデルの分類、metadata、download、thumbnail 「Checkpoint単体」ではなく「Model Bundle」としてAnima/Qwen/Wanの本体+moduleを束ねる。

6. 重要リスク

リスク 重大度 内容 対策
AGPL-3.0 高 Neo runtimeを同梱・改変・配布する場合、AGPL対応が必要。Electron本体へNeoコードを混ぜるとライセンス境界が曖昧になる。 別プロセス、別runtime、notice/source URL、pinしたcommit、改変patch一覧を管理する。公開前に法務判断。
API差分 高 現行YoitomoshiはA1111互換APIに加えForge独自挙動へ依存している。 Neoではmodule、preset、Low Bits、video、multi-imageなど差分が増える。 API compatibility probeを先に作り、生成payloadを直接移植しない。
拡張互換 中から高 Forge NeoのREADMEは3rd-party extension問題を上流サポート外としている。 open issuesにもControlNet / ADetailer / VRAM / performance系が残る。 MVPでは拡張なしAnima txt2imgを最優先。ControlNetとADetailerは後段で個別smoke。
モデル容量と重複 中 Anima、Qwen、Wan、Flux系は本体、text encoder、VAE、quant版で容量が増えやすい。 --model-ref / --forge-ref-comfy-home を活用し、Model Bundle単位で不足表示する。
Anima license 中 Anima公式は非商用ライセンス。Neo Studio自体が使えても、モデル利用条件は別問題。 モデル本体は同梱しない。Model Libraryでlicense warningを表示する。
既存Forge版の退行 高 同じ設定、同じport、同じruntime配下で試すと現行Forge Studioの安定性を壊す。 別アプリ、別設定ファイル、別port、別runtime。共通化はUI部品と型定義から段階的に行う。

8. 最終提案

Yoitomoshi Forge Neo Studioは、現在のForge Studioを置き換えるものではなく、 「新世代モデルをWebUI風に扱うYoitomoshi派生」として作るのが妥当。 最初のゴールはAnima生成1枚の完全な縦切りに絞る。 その後、Qwen-Image、Wan 2.2、Flux.2-Kleinへ広げる。

実装開始前に作るべき最小成果物は、runtime/forge-neo を手動で立てて、 API互換、Anima payload、module選択、生成結果保存を測る NEO_RUNTIME_SPIKE。 ここがPASSしてからElectron統合へ進めば、現行Forge Studioの安定性を守りながらNeo版を作れる。
