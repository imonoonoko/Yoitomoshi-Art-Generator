Codex Atelier: Lo-Fi Dev Companion

このセッションで出た「Codexの働きを、ポモドーロ付きデスクトップ放置ゲームとして育てる」案を、 実装前に必要な情報、MVP要件、アーキテクチャ、リスク、検証観点まで落としたHTMLレポート。 結論は、Codex本体のペット機能に閉じず、Obsidian/Codex Brain、repo docs、Web調査、セッションログを束ねる ローカル常駐の情報工房として作るのが現実的。

作成日: 2026-05-25 JST 対象: Windows / Codex Desktop / local repos 出力: 要件収集 + MVP設計 推奨: Electron + React + TypeScript

1. 結論

実装判断を先に固定
作るべき最初の形: 「Lo-Fi作業部屋 + 相棒キャラ + ポモドーロ + Codex/Git/テストイベントで育つ工房」のローカル常駐アプリ。 `hatch-pet` はCodex内ペット差し替えの中心ではなく、工房内に置くマスコットや進化素材の生成パイプラインとして使う。
企画の核心: このゲームの最大の価値は、見た目の育成や放置報酬ではなく、 工房を整理するほどCodex自体の次回行動が洗練される自己改善ループにある。 書斎に信頼できる調査サイトが溜まり、作業机に定番手順が残り、検証ラボに失敗パターンが蓄積し、 それらが次のCodex実行時の初手、確認順、報告品質を良くしていく。 既存のObsidian/Codex Brainはこのループの外部ツールではなく、工房の書斎・資料庫・長期記憶棚として統合する。

なぜこの方向か

  • Codex Desktop内のペット表示を外部からリアルタイム制御できる保証がまだない。
  • このセッションでは、ユーザーの関心が「既存ペット枠」から「Codexの働き全体のゲーム化」へ移った。
  • Steam系の参照作は、常駐・集中支援・関係性・収集・自動化を組み合わせている。
  • Codex固有の強みは、整理された情報・手順・検証記録がそのまま次回の作業品質を上げられる点にある。
  • 既にObsidian/Codex Brainがあるため、工房は新しい記憶を乱立させるのではなく、Brainやrepo docsを束ねる操作面にできる。
  • Yoitomoshi周辺はElectron/React/TypeScriptの実績があり、Windows常駐UIを作りやすい。

最初に捨てるもの

  • Codex Desktop本体UIの改造。
  • セッションログ形式への強い依存。必ずアダプタ層で吸収する。
  • 変更行数・ファイル数・作業時間だけを直接報酬にする設計。
  • Steam参照作のUI、キャラクター、音源、固有表現のコピー。

5. プロダクト要件

define-requirements形式

Goal

ユーザーとCodexの開発作業を、集中支援・作業記録・育成/経営ゲームに変換し、 放置していても「進んだこと」が見えるだけでなく、整理された工房が次回Codexの調査・実装・検証の質を上げる デスクトップ常駐ソフトを作る。Obsidian/Codex Brain、repo内docs、調査サイト、作業ログ、検証結果を工房に集約し、 Codexが次に読むべき情報へ迷わず辿れる状態を育てる。

Target Users

  • Codexを長時間のローカル開発や調査に使う個人制作者。
  • 作業記録を楽しく残したいが、手動の日報やタスク管理は続きにくいユーザー。
  • ポモドーロ、Lo-Fi、作業同席、デスクトップ常駐ゲームが好きなユーザー。

Success Criteria

  • CodexセッションとGit/テスト結果から、意味のある作業イベントを自動抽出できる。
  • 25分作業後、部屋/工房/相棒キャラの状態が自然に進む。
  • 書斎・作業机・検証ラボの整理状態が、次回Codexの推奨ソース、実行手順、確認観点として再利用される。
  • Obsidian/Codex Brainやrepo docsを「どこに何があるか分かる棚」として見せ、必要な時に正しい一次情報へ誘導できる。
  • 無駄な変更や行数稼ぎを促さず、品質イベントを報酬化する。
  • 設定やリポジトリを壊さず、最初は読取中心で動く。

Non-Goals

  • Codex Desktop本体の未公開内部UIを直接改造する。
  • Steam参照作のキャラクター、UI、音源、固有表現をコピーする。
  • ユーザー承認なしにCodex設定、AGENTS、repoファイルを書き換える。
  • コード量やコマンド数を主報酬にして、作業品質を悪化させる。
  • 整理された工房を、現在性確認なしの固定ルールとしてCodexに押し付ける。
  • Obsidian/Codex Brain、repo docs、Webソースのsource of truth境界を曖昧にする。

tests_failed command exit code / error summary 小トラブル、修理タスク incident生成 失敗自体を罰しすぎない。復旧までセットで扱う。
bug_fixed final summary / diff / test retry 修理完了、耐久度上昇 Quality +4、reputation +1 同じエラーを雑に再試行しただけでは加算しない。
docs_updated docs diff / HTML report / README 図書室、知識庫、掲示板 Knowledge +2 生成物だけでなく内容の種類を分類する。

重要: 報酬は「手数」ではなく「検証された進捗」に寄せる。変更行数、ファイル数、実行コマンド数、長時間実行だけを報酬にすると、 ゲームのために開発品質が落ちる。

10. MVP Scope

最初の完成形

MVPに入れる

  1. ポモドーロタイマー、作業/休憩状態、セッション履歴。
  2. 相棒キャラ1人、部屋1つ、工房レベル、信頼度。
  3. CodexセッションJSONLを読んで、調査/実装/検証/docs/レポートをイベント化。
  4. Git diffのファイル種別分類。
  5. 今日の成果とゲーム進行を表示するダッシュボード。
  6. 調査で使ったサイトをtopic別の書斎棚に保存し、次回の探索候補として出せる。
  7. Obsidian/Codex Brainとrepo docsの索引を読取専用で表示し、情報の集約スペースとして使える。
  8. hatch-pet 既存ONOKO素材を読み込める任意スロット。
  9. HTML daily report export。

MVPから外す

  1. 音声付きフルサウンドノベル。
  2. Codex Desktopの内部ペット表示の動的差し替え。
  3. Codex SDKによる自動実行制御。
  4. 多数キャラ、Live2D、複雑な部屋編集。
  5. 配信向け音源管理。
  6. Brain wikiへの自動書き込み。MVPでは集約・表示・導線までに留める。
  7. クラウド同期、ランキング、公開販売。
MVPの評価基準: 1日のCodex作業後にアプリを開くと、「何を調べ、何を作り、何を検証し、どこで詰まり、何が育ったか」が 3分で分かること。ゲームとしては、次のポモドーロを始めたくなる程度の反応があれば十分。

13. 受け入れ条件

MVPの合格ライン

Functional

  • ユーザーが作業時間/休憩時間を設定し、Pomodoroを開始/停止/スキップできる。
  • CodexセッションJSONLからユーザー依頼、assistant要約、tool call、command outputを抽出できる。
  • 抽出イベントがResearch/Code/Test/Docs/Releaseに分類される。
  • 分類結果が工房レベル、資源、信頼度、今日の履歴に反映される。
  • 今日の成果をHTMLでexportできる。

Safety / Privacy

  • 初期状態では .codex とrepoへの書込をしない。
  • raw promptやtool output全文をゲームDBに保存しない設定を持つ。
  • API key、token、secret、passwordらしき値はredactionする。
  • 監視対象フォルダはユーザーが明示的に選ぶ。
  • Codex SDK/exec連携はMVPでは無効。

UX

  • 集中中はアニメーション、通知、会話が控えめになる。
  • 休憩時にだけ会話と成果サマリが前に出る。
  • デスクトップ端に置いても主要作業を隠さない。
  • 相棒キャラのトーンは設定で調整できる。
  • 毎日見ても同じ文面だけにならない。

  • WakaTime/Gource/GitCityに近いが、commit量や時間ではなく、検証済み進捗・再利用可能な知識・信頼ソースを育成資源にする。

  • Gource/GitCity: Git履歴を工房/街/棚として見せる表現。
  • vexp/Engram/Agentver: local-first、source attribution、versioned skills、memoryの信頼度管理。
