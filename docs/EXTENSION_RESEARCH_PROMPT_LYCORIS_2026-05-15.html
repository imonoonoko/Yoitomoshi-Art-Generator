結論

直接導入の優先度は sd-webui-prompt-format の発想をReact側へ取り込む案が最も高い。 stable-diffusion-webui-text2prompt は自然文からDanbooruタグ候補を出す設計がPrompt Helper改善に使えるが、拡張本体はGradioタブとPython/Hugging Face依存が重く、現行UIにはそのまま馴染みにくい。 a1111-sd-webui-lycoris は現行Forge側にLyCORIS/LoHa/LoKr系の処理が既に入っているため、直接追加は重複・競合リスクが高い。

1. stable-diffusion-webui-text2prompt

機能

  • Input Theme と Input Negative Theme から、Danbooruタグを生成するA1111用タブ拡張。
  • READMEでは、Danbooruタグ説明文の埋め込みと入力文のcosine similarityでタグを選ぶと説明されている。
  • Cutoff and Power / Softmax、Top-k / Top-p、最大タグ数、重み付き選択などを持つ。
  • 実装上は scripts/main.py がGradio UIを作り、WDLike がHugging Faceモデルをロードして推論する。

Yoitomoshiへの適合

  • 既存の PromptHelperPanel はローカル規則・Prompt Library・履歴レビュータグを使って候補を追加する設計。
  • text2prompt本体をForge拡張として入れても、YoitomoshiのReact UIから自然に触れない。
  • 48MB級の事前計算DBやTransformerロードが入り、起動時間・環境依存・GPU/CPUメモリ面の負担が増える。
  • 採用するなら「Danbooruタグ辞書 + 類似候補ランキング」の発想だけを軽量なPrompt Helper候補として取り込む。

実装案

P1候補として、Prompt Helperに「テーマ語からタグ候補」を追加する。初期実装はHugging Face依存を避け、 既存のPrompt Library、レビュー済み正解タグ、PixAI/WD14比較結果からローカル辞書を作る。 将来必要になった場合だけ、埋め込み検索を任意機能として分離する。

2. sd-webui-prompt-format

機能

  • txt2img / img2img / Hires prompt / Negative prompt を対象に整形する。
  • 余分な空白・カンマ、括弧まわり、ピリオド前後、行末カンマを補正する。
  • 同一行内の重複タグ削除、アンダースコア除去、タグalias、Booruサイトからの貼り付け整形を持つ。
  • 実装は javascript/prompt_format.js 中心で、LoRAタグや AND / BREAK / ADDROW / ADDCOL を保護してから整形している。

Yoitomoshiへの適合

  • これはForge側へ入れるより、ReactのPrompt欄に直接入れる方が相性がよい。
  • 既存の promptAppend()、Prompt Helper、History review、Tagger比較とシナジーがある。
  • 「保存前に整形」「生成前に確認」「貼り付け時だけ整形」を選べると、初心者にも扱いやすい。
  • 自動整形は意図しないプロンプト変更が起きるため、初期値は手動ボタンまたはPreflight警告が安全。

最優先の取り込み案

P0候補として、YoitomoshiのPrompt入力まわりに「整形」アクションを追加する。 最小範囲は、空白/カンマ正規化、同一行重複削除、<lora:...> 保護、AND/BREAK保護、除外タグリストの維持。 その後、Taggerのblacklistや履歴レビューの除外タグをPrompt Formatのalias/除外設定へ接続できる。

3. a1111-sd-webui-lycoris

機能

  • 旧A1111向けにLyCORISをExtra Networksとして登録し、<lyco:MODEL:WEIGHT> 形式で利用できるようにする。
  • --lyco-dir でモデル格納先を指定でき、デフォルトは models/LyCORIS。
  • APIとして /sdapi/v1/lycos と /sdapi/v1/refresh-lycos を追加する実装がある。
  • README上も sd-webui < 1.5.0 とForge legacy用途であることを明記している。

ローカルForge確認

  • runtime\forge\webui\extensions-builtin\sd_forge_lora\preload.py に --lyco-dir / --lyco-dir-backcompat がある。
  • runtime\forge\webui\backend\patcher\lora.py に lokr / loha のpatch処理がある。
  • runtime\forge\webui\packages_3rdparty\webui_lora_collection と comfyui_lora_collection にLyCORIS系処理が入っている。
  • そのため、拡張を追加すると旧ローダーとForge内蔵ローダーが重なり、モデル一覧・prompt syntax・infotext変換が混乱する可能性がある。

導入しない理由

A1111 1.5.0のリリースノートではLyCORIS拡張が扱っていたネットワーク種別をLoRA拡張へ統合したと説明されている。 LyCORIS本体READMEも、1.5.0以降は models/Lora または models/LyCORIS に置いて <lora:filename:multiplier> を使う流れを示している。 YoitomoshiではForge側へ旧拡張を入れるより、LoRA/LyCORISモデルを安全に検出・分類・説明するUIを整えるべき。

プロンプト欄に手動の整形ボタンを追加。LoRAタグ、BREAK/AND、Regional Prompter系キーワードを保護しながら空白・カンマ・重複を整える。

低リスク 即効性あり React実装

P1: Theme to Tags

text2promptの発想を、Prompt Library + 履歴レビュー + Tagger正解タグのローカル辞書で軽量化。最初は埋め込み検索なしで候補提示する。

設計必要 既存資産活用

P1: LyCORIS UX

旧 <lyco:> prompt を検出して <lora:> に案内。モデルライブラリではLoRA/LyCORIS/LoCon/LoHa/LoKrを説明付きで表示する。

検証必要 直接拡張不要

採用しない選択肢

案 採用しない理由
3拡張をForgeへそのままclone Gradio UI前提の拡張はReact UIから扱いにくい。起動・依存・設定・DOMが増え、API主導の安定性を落とす。
text2promptのTransformer依存をすぐ組み込む モデルダウンロード、デバイス差、起動時間、メモリ消費が増える。まずローカル辞書ベースで価値を確認すべき。
LyCORIS legacy extensionを同梱Forgeへ追加 Forge内蔵LoRA/LyCORIS処理と責務が重なる。モデル適用の不具合時に原因分離が難しくなる。
