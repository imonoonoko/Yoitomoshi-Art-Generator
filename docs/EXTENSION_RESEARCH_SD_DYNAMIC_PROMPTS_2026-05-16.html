sd-dynamic-prompts 調査レポート

adieyal/sd-dynamic-prompts について、一次情報と二次情報を分けて確認し、 Yoitomoshi Art Generator に直接導入するべきか、React/Electron 側の機能として吸収するべきかを整理した。

作成日: 2026-05-16 JST 対象: A1111 / Forge 系 Dynamic Prompts 前提: Yoitomoshi は Forge API + React UI 判断軸: 探索効率 / 再現性 / 初心者導線 / 安定性

結論

拡張機能本体を Forge にそのまま入れるより、Yoitomoshi 側に「Dynamic Prompt Lab」として取り込む方が有利。 具体的には、{A|B|C}、{2$$A|B|C}、0.5::tag、__wildcard__ の主要構文を React/Electron 側でプレビュー・検証・展開し、既存の Variation / History / Prompt Helper / Prompt Library に接続する。

直接導入は、Gradio UI 前提、Python ライブラリ版数差、Forge 互換、Magic Prompt の大容量モデル、外部 API 機能が絡む。 Yoitomoshi の現在方針である「Gradio UI を埋め込まず、React UI から Forge API を操作する」設計とは相性が弱い。

一次情報から分かること

項目 確認内容 Yoitomoshi への意味
基本目的 README は、A1111 WebUI 用に「random / combinatorial prompt generation」と深い wildcard directory を提供する拡張と説明している。 プロンプト探索・量産・比較のための機能。生成品質を直接上げる拡張ではなく、探索効率を上げる拡張として扱う。
導入方法 WebUI の Extensions tab から導入、または extensions/sd-dynamic-prompts へ clone。Wildcard は拡張フォルダの wildcards 配下。 Yoitomoshi では Forge 拡張フォルダへ直接入れると、API専用起動・無効化管理・配布サイズ・環境差が増える。ユーザーデータ領域に独自保存する方が安全。
主要構文 {A|B|C}、重み、複数選択、ネスト、__wildcard__、glob、recursive glob、YAML/JSON、変数、template parameter、コメントをサポート。 最初から全構文対応は不要。初心者の利便性に直結する subset を UI 化し、上級構文は読み取り専用・後回しにする。
Combinatorial 組み合わせを全列挙できるが、公式 README も Max Generations による制限に触れている。 UI では必ず予測件数と上限を表示する。既定は 4 / 8 / 16 枚程度、無制限実行は入れない。
Seed README は Dynamic Prompts 有効時の random seed、fixed seed、unlink seed from prompt、variation seed の挙動を分けて説明している。 画像 seed と prompt 展開 seed を分離して保存する必要がある。比較では「同一画像seedでpromptだけ変える」導線が重要。
Magic Prompt Prompt 生成モデルを使い、初回に約500MB級モデルをDLする場合がある。低VRAMでは CUDA error の注意もある。 現段階の採用対象外。Yoitomoshi の Prompt Helper はローカル辞書・履歴レビュー・Prompt Library を使う軽量路線を維持する。
I'm feeling lucky Lexica.art API から prompt を取得する機能。 外部API依存・再現性・サービス状態の揺れがあるため、ローカルアプリの既定機能にはしない。
Jinja2 公式ガイドでも上級者向け・スクリプトに慣れたユーザー向けとして扱われている。 UI主体の初心者導線には不向き。安全なテンプレート subset を先に入れ、Jinja2 は実装しない。

二次情報から見える実利用

初心者は構文を覚える前に詰まる

romptn は {A|B|C}、重み、複数選択、Wildcard、Prompt Magic を画像例付きで説明している。 記事の価値は「何ができるか」を日本語で見せる点にあり、実装では syntax 入力欄より、ボタンとプレビューが必要だと分かる。

落とし込み: 「服装だけランダム化」「背景だけランダム化」「表情を重み付きで混ぜる」をテンプレートカード化する。

実用用途は同じ構図の差分探索

romptn のワイルドカード記事は、同じ構図のまま髪型・服装・背景を変える用途を強調している。 Aiarty も dress / hair style / hat / topwear のような要素差分を例にしている。

落とし込み: Prompt Helper と Prompt Library から「可変にするタグ」を選び、固定タグと可変タグを分ける UI を作る。

ControlNet 併用が重要

Aiarty は、服装だけ変えてポーズや髪型を保ちたい場合に ControlNet を使うと説明している。 これは Yoitomoshi の ControlNet Builder と相性がよい。

落とし込み: Dynamic Prompt Lab の比較実行に「構図固定: ControlNetを準備」ボタンを置く。

失敗談は互換性とパス解決に集中

Reddit では Forge 環境で parse error が出る例、Wildcard の subfolder 認識、No values found、hires fix と dynamic prompt の整合性などが見える。 これらは単発報告なので仕様根拠ではないが、UX上の警告設計には使える。

落とし込み: 生成前に missing wildcard、構文エラー、展開件数、hires/upscale時の再展開有無を明示する。

導入案の比較

案 メリット リスク 判断
A. Forge に拡張本体を直接導入 既存実装をほぼそのまま使える。Wildcard Manager や Magic Prompt も揃う。 Gradio UI 前提。API専用Forge起動では UI 資産が活かしにくい。dynamicprompts ライブラリ版数差・pyparsing・Magic Prompt依存が増える。 非推奨 実験環境なら可。本体配布の既定にはしない。
B. Python dynamicprompts を Electron IPC から呼ぶ 公式ライブラリに近い構文互換を得やすい。 Forge Python / venv / package更新 / Windows path / subprocess コストが増える。UIプレビューのたびに重い。 後回し 高度構文互換が必要になった時だけ検討。
C. TypeScript で安全 subset parser を実装 既存 Prompt UI、Variation、History、Workspace と自然に接続できる。プレビュー・検証が高速。配布しやすい。 全構文互換ではない。parser とテストを書く必要がある。 推奨 Yoitomoshi ではこの案を採用する。
D. Prompt Library のみで疑似ランダム化 実装が軽い。既存資産を活用できる。 ユーザーが外部 wildcard 資産や {A|B|C} を使えない。比較生成の価値が弱い。 補助 C案のデータ源として使う。
E. Magic Prompt / Lexica API 連携 短いpromptから長い候補を出せる。 大容量モデル、VRAM、外部API、出典不明語、再現性低下。既存のローカル Prompt Helper 方針とずれる。 非採用 まずはローカル補助に限定する。

推奨する落とし込み

1. Dynamic Prompt Lab

PromptPanel 内に折りたたみパネルとして追加。トップレベルタブは増やさない。

  • 現在の Prompt から variants を検出
  • 構文エラーと missing wildcard を表示
  • 展開プレビューを 8 件表示
  • 画像 seed と prompt seed を別表示

2. Builder UI

ユーザーに構文を直接覚えさせず、チップ編集から variants を作る。

  • 選択タグを「候補グループ化」
  • 重みスライダーを 0.5::tag に変換
  • 「2個選ぶ」stepper を 2$$ に変換
  • 固定タグ / 可変タグを視覚的に分ける

3. Wildcard Library

拡張フォルダではなく userdata/dynamic-prompts/ に保存する。

  • Prompt Library サブカテゴリから wildcard 作成
  • History review accepted tags から wildcard 作成
  • txt / yaml / json import
  • 重複・空行・危険な path を検証

4. Variation 連携

既存 VariationPanel に axis prompt を追加する。

  • 2 / 4 / 8 枚の prompt 展開比較
  • 同一画像seedで prompt だけ変える
  • 結果に resolved prompt と template を保存
  • 良い候補を QuickPreset / Workspace へ保存

5. History / Workspace

後から再現できるよう、展開前と展開後の両方を保存する。

  • dynamicTemplate
  • resolvedPrompt
  • dynamicSeed
  • wildcardSources

6. Prompt Helper

自然文から「どこをランダム化するか」を提案する。

  • 服装差分
  • 髪型差分
  • 背景差分
  • 表情の重み付き差分

リスクと対策

リスク 対策
組み合わせ爆発で大量生成される 常に推定件数を表示し、既定 8、上限 100 未満。無制限は実装しない。
同じ seed なのに結果が再現できない 画像seedとprompt展開seedを別に持つ。履歴とWorkspaceに両方保存する。
Wildcard file が見つからない 生成前 preflight で missing wildcard を止める。候補パスと作成ボタンを出す。
LoRA / Regional Prompter / BREAK / AND を parser が壊す <lora:...>、AND、BREAK、Regional Prompter token は protected token として扱う。
外部 wildcard pack の品質が低い import 後にタグ数、重複、長すぎる行、NSFW/不要語、空行を表示。既定では Prompt Library 由来を優先。
Magic Prompt が重い / 外部依存が増える 初期実装では採用しない。既存 Prompt Helper のローカル候補生成を強化する。
YAML/JSON import の安全性 構造データとして parse し、path traversal と巨大ファイルを拒否。任意コード実行系テンプレートは扱わない。
